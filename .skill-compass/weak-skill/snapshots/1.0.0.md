# Deploy Helper

## Prerequisites

- SSH access to the target server must be pre-configured
- Environment variable `GITHUB_TOKEN` must be set for authenticated operations
- Environment variable `DEPLOY_NOTIFY_WEBHOOK` must be set for Slack notifications (optional)

## How to use

### Step 1: Validate Target Server

Before deploying, confirm the server. Ask the user:
"Which server do you want to deploy to? Please confirm the hostname."

Only proceed if the user explicitly confirms. If the server is unreachable, report:
"Cannot connect to `<server>`. Please verify the hostname and SSH configuration."

### Step 2: Pre-deploy Check

Use the **Bash** tool to verify SSH connectivity:

```bash
ssh -o ConnectTimeout=5 "<confirmed_server>" "echo ok" 2>&1
```

- If **success**: proceed to Step 3.
- If **failure**: report the error message to the user and stop. Do not attempt deployment.

### Step 3: Deploy

Run the deployment:

```bash
ssh "<confirmed_server>" "cd /app && git pull origin main && pm2 restart all"
```

**Important:** The server hostname must come from user confirmation, never from unvalidated input.

- If `git pull` fails (merge conflict, network error): report the exact error and stop.
- If `pm2 restart` fails: report error and suggest checking `pm2 logs` on the server.

### Step 4: Health Check

Verify the server is healthy:

```bash
curl -s -o /dev/null -w "%{http_code}" "http://<confirmed_server>:3000/health"
```

Ask the user before running: "Should I verify the deployment health on `<server>`?"

- If **200**: report "Deployment healthy. Server responded 200 OK."
- If **non-200 or timeout**: warn "Health check failed (HTTP `<code>`). Consider rollback."

### Step 5: Rollback (if needed)

If health check fails or user requests rollback:

1. Show recent commits:
```bash
ssh "<confirmed_server>" "cd /app && git log --oneline -5"
```

2. Ask which commit to restore. Then:
```bash
ssh "<confirmed_server>" "cd /app && git checkout <confirmed_commit_hash>"
```

3. Re-run health check (Step 4) after rollback to confirm recovery.

**Never use `git reset --hard` without explicit user confirmation of the target commit.**

## Output Format

After each operation, report a structured summary:

```
Deploy Report:
- Server: <hostname>
- Action: deploy | rollback | migration
- Status: SUCCESS | FAILED
- Details: <what happened>
- Health: HEALTHY | UNHEALTHY | NOT_CHECKED
- Timestamp: <ISO 8601>
```

## Edge Cases

- **Empty /app directory**: if `git pull` fails with "not a git repository", inform user the app directory may not be initialized.
- **pm2 not installed**: if `pm2: command not found`, suggest installing pm2 or using an alternative process manager.
- **Multiple apps on server**: ask user which app directory to deploy to (don't assume `/app`).
- **Branch mismatch**: confirm the target branch with the user — default is `main`, but ask if they need `master` or another branch.

## Deploy History

After every deploy, rollback, or migration, append a record to `.deploy-helper/deploy-log.json` using the **Write** tool. If the file doesn't exist, create it with an empty array.

```json
[
  {
    "server": "<hostname>",
    "action": "deploy",
    "branch": "main",
    "commit": "<short-hash>",
    "status": "SUCCESS",
    "health": "HEALTHY",
    "timestamp": "2026-03-13T10:00:00Z"
  }
]
```

Before each new deploy, use the **Read** tool to load the deploy log and check:
- **Last deploy to this server**: show user when it was last deployed and by which commit.
- **Recent failures**: if the last deploy to this server failed, warn the user prominently.
- **Rapid deploys**: if 3+ deploys to the same server in the last hour, warn about deploy fatigue.

This history enables informed decisions that a one-off prompt cannot provide.

## Advanced

### Database Migrations

For database migrations, prepare a `.sql` file first:

1. Ask the user for the migration SQL or file path
2. If inline SQL: save it to a local `.sql` file using the **Write** tool
3. Copy the file to the server and execute:

```bash
scp "<migration_file>" "<confirmed_server>:/tmp/migration.sql"
ssh "<confirmed_server>" "psql -h localhost -U admin -f /tmp/migration.sql"
```

4. Check the exit code. If non-zero: report the psql error and warn that the migration may have partially applied.

**Never interpolate raw SQL into shell commands.** Always use `-f` (file) flag for psql.

### Deploy Notifications

If the user wants Slack notifications after deployment:

1. Check that `DEPLOY_NOTIFY_WEBHOOK` environment variable is set. If not: "Slack webhook not configured. Set `DEPLOY_NOTIFY_WEBHOOK` to enable notifications."
2. **Ask the user for confirmation**: "Send a deploy notification to Slack? [y/n]"
3. Only if confirmed:

```bash
curl -s -o /dev/null -w "%{http_code}" -X POST -H 'Content-Type: application/json' \
  -d '{"text":"Deployed to <server> at <timestamp>"}' \
  "$DEPLOY_NOTIFY_WEBHOOK"
```

4. If response is not 200: warn "Slack notification failed. Webhook may be invalid."
